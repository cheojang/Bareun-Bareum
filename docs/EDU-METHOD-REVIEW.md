# 교육 방법론 진단 결과 (2026-07-03)

훈련 파이프라인 전체(분석 → 훈련 처방 → 연습 → 복습 스케줄링)를 코드 레벨로 점검한 결과.
각 항목은 근거 파일과 개선안을 포함한다. 우선순위: 🔴 높음 → 🟠 중간 → 🟡 낮음.

## ✅ 1. 발달연령을 무시한 "교정 필요" 판정 — 구현 완료 (2026-07-03)

**구현 내용:**
- `src/lib/developmental-norms.ts` 신설 — `getChildAgeMonths(birthDate)`,
  `getDevelopmentalStatus(phoneme, ageMonths)`(developing/emerging/expected/unknown),
  `getDevelopmentalDisplay()`(배지·안내 문구)
- `error-analysis` 라우트: 아이 나이×목표 음소 습득 시기 비교 후
  ① 응답에 `developmental` 필드 추가 ② 약점 집계(`recalculateWeakPhonemes`)에서
  발달상 정상 음소는 "집중교정필요"로 승급 차단 (developing→관찰중, emerging→꾸준한연습필요 상한)
- DB `weaknessLevel`이 단일 소스라, 리포트·데일리미션·최소대립쌍 자동선택 등
  모든 읽기 화면이 게이트를 자동 승계 (기존 레코드는 다음 분석 시 자기치유)
- `AnswerNoteClient`: 발달 안내 배너 추가 — "지금 못 하는 건 자연스러운 발달 과정"
  맥락을 교정 콘텐츠보다 먼저 제시해 부모 불안 완화

**원래 문제 (참고용 보존):**

**문제:** 음소 습득 연령 데이터가 코드에 정의되어 있으나 어디서도 사용되지 않음.

- `src/lib/korean-phonetics.ts`의 `ACQUISITION_ORDER`: ㅅ/ㅆ/ㅈ/ㅉ/ㅊ는 4~5세,
  ㄹ은 5~6세 습득이 정상 — 정의만 있고 참조하는 코드 0곳 (`getExpectedAgeForPhoneme`도 미사용)
- AI-LOGIC.md 명세의 "나이별 응답 기준"(만 3세 미만 → 훈련법 미제공, 3~4세 → 1~2단계만)이 미구현
- `gemini-client.ts`의 `calcChildAge`는 생일 미입력 시 **무조건 4세로 가정**

**결과:** 3세 아이가 ㄹ을 못 하는 것은 정상 발달인데 "오류율 85%, 집중 교정 필요"
핑크 배지 + 4단계 훈련법을 그대로 처방 → 부모 불안 조장, 발달적으로 부적절한 조기 훈련 유도.

**개선안:** 약점 판정·훈련 처방 전에 `아이 나이 vs ACQUISITION_ORDER` 비교 분기 추가.
습득 연령 전이면 "아직 배우는 중인 소리예요. 지켜봐도 괜찮아요 🌱" 배지로 다운그레이드.
데이터·나이 계산 함수가 이미 있어 구현 비용 낮음.

## 🟠 2. 성공 판정이 100% 부모 주관

**문제:** 연습·복습 모두 부모의 "잘 됐어요/아직 어려워요" 버튼이 유일한 판정 입력
(`PracticeClient.tsx`, `ReviewClient.tsx`). 자기 아이 발음에 익숙한 부모의 귀는 오류를
잘 못 듣는다는 것이 조음치료의 알려진 함정. 이 판정이 SM-2 스케줄링·적응형 난이도·
약점 통계의 유일한 입력이라 오차가 전 시스템에 전파됨.

**개선안:**
- 녹음 훅(`usePracticeRecorder`)이 이미 있음 → 판정 전 "아이 녹음 ↔ TTS 모범 발음"
  A/B 재생 비교 유도
- 주 1회 "지난주 녹음 vs 오늘 녹음" 비교 카드로 부모 귀 보정

## 🟠 3. SM-2(암기용 알고리즘)를 운동 학습에 적용

**문제:** `src/lib/sm2.ts` — SM-2는 선언적 기억(어휘 암기)용. 조음은 운동 학습(motor
learning)이라 원리가 다름. 14일 상한(MAX_INTERVAL_DAYS)은 좋은 보정이지만,
**"같은 단어 카드 5회 성공 = 졸업(isLearned)"** 이 위험 — 조음치료의 핵심 난제는
*일반화*(연습 단어 성공 ≠ 새 단어·자발화 전이)인데 이를 검증하지 않고 졸업시킴.

**개선안:** 졸업 판정에 **일반화 프로브** 추가 — 같은 음소·같은 위치의 *미연습* 단어
2~3개를 제시해 통과해야 졸업. `getSimilarPatternWords()`로 신규 단어 선정 가능.

## 🟡 4. 훈련 위계가 "가장 어려운 것부터" 시작

**§4-1 성공 우선 세션 배치 — 구현 완료 (2026-07-03):**
- `PracticeClient.buildCycleItems`: 첫 사이클에 한해 "쉬운 같은-음소 유사단어"를
  워밍업 카드로 맨 앞에 배치하고, 가장 어려운 오답단어를 그 뒤로 미룸
  (wordInfos의 difficulty==="easy" 단어 선택, 없으면 워밍업 없이 기존 순서)
- 워밍업 단어는 해당 사이클 유사단어 목록에서 중복 제거해 한 번만 노출
- 2번째 사이클부터는 이미 몸이 풀린 상태이므로 기존 순서 유지
- 워밍업 카드에 "🌱 워밍업 · 쉬운 소리부터" 배지 표시
- §4-2(음절 카드), §4-3(듣고 고르기 변별)은 미착수

**(원래 진단 내용)**


**문제:**
- 전통 조음치료 위계: 청지각 변별 → 독립음 → 음절 → 단어 → 문장. 앱은 stage1을
  **"틀렸던 단어"** 로 시작 — 아이가 제일 못 하는 자극으로 세션을 여는 구조.
  코드 주석의 설계 철학("80% 성공률 유지가 최적 동기 구간", `adaptive-difficulty.ts`)과 모순
- 음절 단계·청지각 변별 단계 부재 (Gemini 훈련법 텍스트에는 "소리→음절→단어"가

**개선안:**
1. 세션 시작을 easy 유사패턴 단어 1~2개 워밍업으로 열고 오답 단어를 뒤로 배치
2. TTS로 음절 카드("사–", "스–") 단계 추가 (저비용)
3. 듣고 고르기 게임 — "방금 들린 게 '사과'일까 '따과'일까?" — 산출 전 변별 훈련.
   TTS+이미지 인프라 재사용 가능

## 🟡 5. 청각 폭격(auditory bombardment)의 노출량 부족

**문제:** `PracticeClient.tsx`의 `AuditoryBombardment` — 오답+유사 단어 최대 6개를
각 1회, 1초 간격 재생 후 종료. 실제 기법은 목표 음소 단어 15~20개 반복 노출
(아이는 듣기만).

**개선안:** 목표 음소 단어 풀에서 12개 내외 × 2회전 재생 + "따라 말하지 않고
듣기만 해요" 안내 문구.

## 🟡 6. "오류율" 지표의 통계적 왜곡

**§6-1 문구 정정 — 구현 완료 (2026-07-03):**
- `comprehensive` 리포트: 등급 설명 "오류율 30% 이상" → "최근 오답 중 30% 이상",
  조음 발달 현황 안내문에 "발음 실패율이 아니라 오답 중 비중" 명시
- `daily-missions`: "약점이에요 (오류율 X%)" → "최근 오답에 가장 자주 나와요 (오답 중 X%)"
- `gemini-client` 약점 리포트 프롬프트: errorRate/totalAttempts의 실제 의미(오답 분포·
  총 오답 건수)를 명시하고 "오류율" 표현 사용 금지 지시 → Gemini가 과장 표현 되풀이 방지
- (progress·report 페이지는 %가 아닌 "N회 오류" 카운트/바만 표시 → 이미 정확, 미변경)

**§6-2 성공률 기반 지표 전환은 미착수** (아래 개선안 2, 규모 중~대):

**문제:** `오류율 = 음소별 오류 수 ÷ 전체 오답 수` (AI-LOGIC.md, weak-phonemes API).
부모가 **오답만 입력**하는 구조라 분모에 성공 시도가 없음 — 아이가 ㅅ를 100번 잘
